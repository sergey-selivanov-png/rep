### Принципы построения системы

#### Вертикальный шардинг

Принцип: Разделение таблиц базы данных по физическим серверам по типу данных, чтобы оптимизировать доступ и обработку.

### Сервер 1: Таблицы пользователей
users
profiles (дополнительная таблица с информацией о профиле пользователя)

### Сервер 2: Таблицы книг и их метаданные
books
authors (таблица с информацией о авторах)
genres (для указания жанров)

### Сервер 3: Таблицы магазинов и транзакций
stores
transactions (таблица для хранения информации о покупках)

Преимущества:

Изоляция нагрузки между различными типами данных.
Оптимизация по паттернам доступа к различным данным.
Увеличение производительности при параллельной обработке данных.

#### Горизонтальный шардинг

Принцип: Разделение одной таблицы по записям (строкам) между несколькими серверами, что позволяет обрабатывать большие объемы данных.

Сервер 4: Шард 1 — книги с book_id от 1 до 100,000 (например, жанры: фантастика, детективы).
Сервер 5: Шард 2 — книги с book_id от 100,001 до 200,000 (например, жанры: классика, научпоп).
Сервер 6: Шард 3 — книги с book_id от 200,001 до 300,000 (например, жанры: бизнес, учебники).

Преимущества:

Линейное масштабирование при росте данных.
Параллельная обработка запросов.
Улучшенная доступность данных.

### Режимы работы серверов

#### Серверы для вертикального шардинга:
Серверы (1, 2 и 3) обслуживают разные типы данных (пользователи, контент, транзакции).
Каждый сервер имеет свою нагрузку и оптимизирован для конкретного типа запросов, что повышает производительность.

#### Серверы для горизонтального шардинга:
Серверы (4, 5 и 6) хранят данные о книгах, разделенные по диапазонам book_id.
Это архитектурное решение позволяет обрабатывать более крупные объемы данных и увеличивает скорость выполнения запросов.
